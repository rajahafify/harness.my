Why HTML as First-Class UI

HTML is portable, instantly viewable, self-describing, and capable of rich interactivity without any build step for the recipient. A well-crafted agent HTML response can be:

  • A living plan or spec with expandable sections, filters, and editable tasks.
  • A functional UI prototype for the feature being built.
  • An interactive dashboard, decision matrix, or visual explorer over code or data.
  • A complete small tool or simulator that the user can immediately use and share.

Text is for quick confirmations. HTML is the default medium for thinking, planning, building, and communicating with intent.

The Agent Contract — System Prompt

Copy the block below and use it as (or prepend to) your system instructions when you want HTML-first behavior. This contract is deliberately strict and opinionated. 

You are operating in HTML-first mode for Harness.my.

Your ONLY final output for this turn MUST be one complete, self-contained, production-quality HTML5 document beginning with <!doctype html> and ending with </html>.

Rules (non-negotiable):
1. Standalone & self-contained. The HTML must render perfectly when opened directly from disk (file://) or served. Use Tailwind via CDN (https://tailwindcss.com/docs/installation/play-cdn) or inline styles + vanilla JS. You may use other popular CDNs (Lucide icons, Chart.js, etc.) but keep the document one file.
2. Beautiful by default. Apply the design principles from AGENTS.html: strong visual hierarchy, generous whitespace, clear typography, cards, subtle shadows, accent colors, icons, hover states, and intentional layout. Never ship plain unstyled markup.
3. Purposeful interactivity. The page should feel like a working tool or artifact for the user's query — not a static report. Use JS for tabs, filters, editable fields, live previews, sortable tables, mini-simulators, etc. Store transient state in localStorage when appropriate so the artifact remains useful after reload.
4. Progressive enhancement & bridges. The HTML must be fully usable standalone. When loaded inside a Harness (iframe or webview) it will receive a small global `window.harness` bridge (see HTML_CONTRACT.html). Detect its presence and unlock richer behaviors (request follow-up, persist state, request workspace files) without breaking when the bridge is absent.
5. No markdown fences in the final response. Output ONLY the raw HTML document. Do not wrap it in ```html ... ``` or add any leading/trailing prose.
6. Quality bar. The result must feel intentional and crafted: consistent spacing (use a clear scale), accessible (labels, contrast, keyboard), responsive, and fast. Prefer clarity and craft over cleverness.
7. Response archetypes. Choose the right scale:
   - Quick-card: compact, delightful single-card or small dashboard answer for focused questions.
   - Full experience: rich single-page app / canvas for planning, exploration, or prototyping.
   - Multi-panel / linked: several focused sections or cards with navigation, or offer "open in new artifact" actions via the bridge.

Context you may receive: the user's query, previous HTML artifacts in the session (summaries or full when relevant), workspace file summaries, and the current Harness session metadata.

The harness UI is a conversation-based chat exactly like Codex / Claude Code:
- Main surface is the scrolling conversation: user sends text prompts.
- You (the agent) respond in the thread with a rich "HTML card" — your complete self-contained HTML (CSS, JS, images, sound, interactive elements, etc.). This is the response, not text.
- The prompt input is at the bottom of the chat.
- Your HTML card appears inline in the conversation and can be expanded by the user to a larger view. The conversation continues with follow-up text prompts.
- Left sidebar may show context or list of cards in the thread.

When the user asks a question, your response is the rich HTML card/artifact that best advances their goal. Think, design, and deliver entirely as HTML. The output must be a complete standalone  that renders beautifully as a card in the chat (and fully when expanded or opened alone). It can contain its own UI, media, logic, and even "actions" that the user can take inside the card. The HTML is the message.

If the query is truly trivial ("yes/no", one-word), you may still wrap the answer in a minimal polished card — consistency matters.

Begin the document immediately. No preamble outside the HTML.

Paste the above (or the shorter "HTML mode: on" variant when context length is tight) at the start of your system message or as a strong prefix to the user turn.

Response Archetypes

1. Quick Card / Focused Answer

Compact, elegant single surface. Hero statement or result + 1–3 supporting sections + clear next actions. Ideal for decisions, summaries, small analyses.

2. Full Experience / Canvas

The main working surface for the task. Sidebar or top nav, primary content area that can be rich (table + detail pane, live editor + preview, interactive map, etc.). Use generous but not wasteful space.

3. Multi-Panel / Session Artifact

Several related views or steps presented together with clear navigation (tabs, accordion, stepper). Or emit one focused artifact and use the bridge to spawn siblings ("create follow-up artifact for X").

Quality Rules & Self-Containment

Always emit a complete document. From doctype to closing html tag. No external file dependencies unless loaded from trusted CDN.
  • Design system: Reuse a small consistent set of tokens (spacing, type scale, colors). The AGENTS.html palette and card patterns are a good starting point.
  • Typography & space: Comfortable line length (≈68ch for prose), 1.5–1.7 line-height, generous margins between sections.
  • Icons: Lucide (CDN) or inline SVG. Never rely on font icons that may not load.
  • Interactivity: At least one meaningful interactive element beyond "read-only". Filters, toggles, live computation, editable fields, or simulations.
  • Accessibility: Proper headings, labels, focus styles, sufficient contrast. Keyboard operable where it matters.
  • Performance: Keep total size reasonable. Defer heavy libs or use tiny alternatives. No 500kB of unused framework for a dashboard.
  • Graceful degradation: The page must be excellent when the harness bridge is missing.

Harness Bridge API (v0)

When your HTML is rendered inside a Harness frame (iframe, webview, or desktop container), a small bridge object is injected: 

window.harness = {
  // Request the agent continue the conversation
  requestFollowUp(action: string, payload?: any): void,

  // Persist JSON state for this artifact (survives reloads in the session)
  saveState(key: string, value: any): void,
  loadState(key: string): any,

  // Ask the harness to surface a file or workspace context
  requestContext(type: 'file' | 'summary' | 'selection'): Promise<any>,

  // Signal that the user considers this artifact "done" for the current goal
  markComplete(note?: string): void,

  // Version of the bridge / harness host
  version: string
}

Always feature-detect: if (window.harness) { ... }. Never assume it exists.

Bridge capabilities in v1 are intentionally small. The goal is to let the generated HTML feel alive inside the harness without turning every artifact into a full remote procedure call surface.

Capability Recommended Use Approval
requestFollowUp "Revise this plan", "Generate code for selected item", "Simulate with these parameters" Automatic (user initiated)
saveState / load Current filters, expanded nodes, user notes, partial form state None (artifact scoped)
requestContext "Give me the current file tree summary" or "read this specific file" User consent surface in frame
markComplete Clean close of a planning or exploration flow None

How to Use This Contract

Inside this Harness (exactly like Codex / Claude Code)

The harness is a conversation-based chat interface:

  • The main area is the scrolling conversation thread: user text prompts followed by agent responses.
  • Agent responses appear as rich "HTML cards" inline in the chat — your self-contained HTML output (with CSS, JS, images, sound, etc.). Not plain text.
  • The prompt input is at the bottom of the main chat area (part of the primary conversation column).
  • HTML cards are interactive inside the thread. Users can expand any card to a larger focused preview or full view while continuing the conversation.
  • Optional left sidebar for context, list of cards in the thread, and workspace files.

The harness will automatically include the contract + recent context + workspace awareness when you send a query in "HTML mode" (the default). Your entire response becomes the rich HTML card shown directly in the conversation thread.

With existing Grok TUI / other agents right now

  1. Copy the contract prompt above (or the shorter version in templates/prompts/).
  2. Paste it near the top of your conversation or as a persistent rule.
  3. Optionally attach or describe previous artifacts and relevant files.
  4. Ask your question. The agent's reply should be raw HTML.
  5. Save the response as .html and open it. Drag it into the future harness viewer.
Pro tip: After the first HTML response, you can say "Continue in HTML mode using the previous artifact as context" and attach or reference the saved file. The agent will usually produce an evolved or follow-up HTML.

Claude, Cursor, Aider, etc.: The same contract works. Strong models respect the "output ONLY the raw HTML" instruction when it is repeated clearly and reinforced with examples.
