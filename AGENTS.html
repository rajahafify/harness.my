AGENTS

Design-Oriented Guidelines for Coding Agents

🎨
This project prefers beautifully designed HTML over Markdown.

All documentation, guidelines, specs, internal tools, and instruction files should feature thoughtful design elements, strong visual hierarchy, modern UI patterns, icons, cards, color accents, and polished presentation — not plain functional markup.

🚀 Evolving: HTML is now first-class UI for agent responses.

Agents respond to queries with complete, self-contained, interactive <!doctype html> documents by default. See HTML_CONTRACT.html — the canonical contract, prompt, patterns, and bridge spec. The contract document itself demonstrates the standard.

Behavioral guidelines to reduce common LLM coding mistakes while producing work that feels intentional and well-crafted. These instructions apply to all AI coding assistants (Grok, Claude, Cursor, and similar) working in this repository.

Think Before Coding

Don't assume. Don't hide confusion. Surface tradeoffs.

Before implementing:

  • State your assumptions explicitly. If uncertain, ask.
  • If multiple interpretations exist, present them — don't pick silently.
  • If a simpler approach exists, say so. Push back when warranted.
  • If something is unclear, stop. Name what's confusing. Ask.
2
✨

Simplicity First

Minimum code that solves the problem. Nothing speculative.

  • No features beyond what was asked.
  • No abstractions for single-use code.
  • No "flexibility" or "configurability" that wasn't requested.
  • No error handling for impossible scenarios.
  • If you write 200 lines and it could be 50, rewrite it.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

3
✂️

Surgical Changes

Touch only what you must. Clean up only your own mess.

When editing existing code:

  • Don't "improve" adjacent code, comments, or formatting.
  • Don't refactor things that aren't broken.
  • Match existing style, even if you'd do it differently.
  • If you notice unrelated dead code, mention it — don't delete it.

When your changes create orphans:

  • Remove imports/variables/functions that your changes made unused.
  • Don't remove pre-existing dead code unless asked.

The test: Every changed line should trace directly to the user's request.

🎯

Goal-Driven Execution

Define success criteria. Loop until verified.

Transform tasks into verifiable goals:

  • "Add validation" → "Write tests for invalid inputs, then make them pass"
  • "Fix the bug" → "Write a test that reproduces it, then make it pass"
  • "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:

1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]

Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.

🎨

Design-Conscious HTML

HTML deliverables should be visually crafted and delightful — not merely functional.

This project values design elements in HTML.

When creating or editing any HTML (documentation, guidelines like this file, internal tools, specs, UIs, reports, landing pages, etc.):

  • Apply deliberate visual design: strong hierarchy, generous and consistent whitespace, clear typographic rhythm, and balanced layout.
  • Use modern UI patterns and design elements: cards, colored callouts, iconography, subtle shadows, badges, dividers, accent colors, grids/flex, hover states, and micro-interactions where they improve the experience.
  • Make content scannable and pleasant to read. Use color, spacing, and visual weight to guide the eye and highlight what matters.
  • Plain unstyled markup is only the starting point. Elevate it with design intent so the result feels professional, intentional, and high-quality.
  • Design is part of clarity. Treat the reader's visual and emotional experience as a core requirement of the deliverable.

Examples of good design elements: nice headers with accent bars, numbered cards, icon + text combinations, success/warning banners, well-styled code blocks, consistent button-like pills, and breathing room between sections.

✅ These guidelines are working if

Fewer unnecessary changes in diffs, fewer rewrites due to overcomplication, clarifying questions come before implementation rather than after mistakes, and HTML artifacts demonstrate care, visual polish, and thoughtful design elements.
