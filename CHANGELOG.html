2026-06-14

UI mock now matches the reference screenshot + fully conversation-based model

After a few rounds of feedback, the harness-ui-mock.html was rewritten to follow the provided Cursor/Claude Code-style screenshot as closely as possible while staying true to the core idea of Harness.my.

The interface is now a proper conversation thread:

  • User types a text prompt in the composer at the bottom of the main column.
  • The agent replies in the thread with a rich, self-contained HTML card — not plain text.
  • Cards are interactive, can contain CSS, JS, images, and sound hooks, and support an "Expand" action for a larger focused view.

Layout closely mirrors the reference image:

  • Left sidebar with New chat, Search, Automations, Projects, and Chats list (selected item highlighted).
  • Main area shows the conversation with structured text, inline HTML cards (styled similarly to the "Edited files" blocks in the screenshot), and small note callouts.
  • Bottom composer is nearly identical in style and placement, with a + button, input, model selector, and Send.
  • Right "Environment" panel includes Changes, Progress checklist, and Sub-agents (adapted to our HTML-card context).

Supporting updates were made to HTML_CONTRACT.html so the agent contract now explicitly describes the conversation + inline HTML card model. The earlier summary document was replaced by this proper prepend-only CHANGELOG.html.

Files touched: harness-ui-mock.html (major), HTML_CONTRACT.html, examples/, templates/, AGENTS.html, README.md, and this changelog.
2026-06-14 (earlier)

Foundation work: HTML-first contract and initial mock iterations

Established the core philosophy that agents in Harness.my default to responding with complete, standalone, beautifully designed HTML documents.

Created the canonical HTML_CONTRACT.html (a self-demonstrating spec) and supporting templates + an early interactive plan example. Initial mock versions explored different layouts (multi-column, prompt in main column) before landing on the conversation-thread model with HTML cards as the agent's voice.

All artifacts follow the project's existing guidelines: intentional design, minimum code that solves the problem, and HTML preferred for important documentation and deliverables.
