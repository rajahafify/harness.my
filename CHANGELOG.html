2026-06-14 18:00

Next development phases defined; UI mock as living spec

With the high-fidelity harness-ui-mock.html now closely matching the reference Cursor/Claude Code screenshot and embodying the conversation-based model (user text prompts in main column → agent replies with rich interactive HTML cards in the thread), we have a solid, intentional visual and interaction specification.

The README and this CHANGELOG have been updated to document the current state, including full commit history with GitHub links and timestamps on all entries. The project remains committed to prepend-only, blogpost-style documentation for the changelog.

Proposed Next Phases (goal-driven, aligned with AGENTS.html principles of simplicity, surgical changes, and HTML as first-class):

  1. Real Interactive Viewer (MVP)
    Scaffold a dynamic web app (minimal Vite + TS + Tailwind or even enhanced static site to stay lightweight) that turns the static mock into a fully functional chat.
    • Dynamic conversation thread: append user messages and agent "HTML card" responses.
    • Card rendering: support full/embedded HTML (initially via srcdoc iframes or sanitized DOM injection for isolation).
    • Core interactions: send prompt, expand cards, basic sidebar navigation for sessions/artifacts.
    Success criteria: User can have a multi-turn conversation where "agent" produces different rich HTML cards based on prompts (start with template library + simple generation).
  2. Real Agent Integration
    Wire the prompt composer to an actual LLM (leveraging existing Grok auth/BYOK patterns or a thin proxy) using the HTML_CONTRACT.html as system prompt + context (previous cards, workspace summaries).
    • Parse agent output: ensure/strip to pure <!doctype html> and render as card.
    • Handle errors gracefully (e.g. non-HTML fallback card).
    Success criteria: Typing a real prompt produces a valid, beautiful HTML card from a live agent that appears in the thread.
  3. Sessions, Persistence & History
    Add multi-session support in the left sidebar.
    • Persist conversations (localStorage first, then file-based or simple backend).
    • Ability to switch sessions, fork conversations, export a thread as a single rich HTML page.
    • Thread metadata: timestamps, agent used, etc.
  4. Harness Bridge for Card Interactivity
    Implement the window.harness API (requestFollowUp, saveState, etc.) so that elements inside the rendered HTML cards can meaningfully interact with the harness (e.g. "Revise this plan", "Open file", persist card-specific state).
    • Secure injection only for cards rendered by the harness.
    • Progressive enhancement: cards still work standalone.
  5. Desktop Experience & Auth
    Wrap the viewer in Electron (or evaluate Tauri for lighter weight) to match the "desktop agentic tool" feel from the original vision.
    • Native menus, window management, deep workspace integration.
    • Full BYOK + login flows (browser OAuth windows) for providers.
    • Optional local server component for advanced features (live reload, file mgmt).
  6. Polish, Dogfooding & Expansion
    Use the harness on itself to generate more example HTML cards, improve the contract, add rich patterns (data viz, code explorers, prototypes), accessibility, performance.
    • Create a gallery of high-quality committed HTML response examples.
    • Iterate based on real usage (e.g. this very project).

These phases are designed to be incremental and verifiable. Each can produce working, intentional artifacts. The static mock remains the reference for visual and interaction fidelity until the real app surpasses it.

This entry itself will be committed as part of documenting the planning step.
